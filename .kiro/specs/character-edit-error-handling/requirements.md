# Требования к исправлению бага обработки ошибок редактирования персонажа

## Введение

Данная спецификация описывает исправление критического бага в Telegram боте D&D, связанного с неправильной обработкой ошибок ввода при редактировании числовых полей персонажа. Проблема заключается в некорректном сбросе состояния FSM при возникновении ValueError, что приводит к прекращению ожидания корректного ввода от пользователя.

## Глоссарь

- **FSM**: Finite State Machine (конечный автомат состояний) - механизм управления состояниями бота
- **ValueError**: Исключение Python, возникающее при попытке преобразования некорректного значения в число
- **CharacterEdit.waiting_for_value**: Состояние FSM, в котором бот ожидает ввод значения для редактирования поля персонажа
- **CharacterCreation.editing_sheet**: Состояние FSM для отображения меню редактирования персонажа
- **process_edit**: Функция-обработчик сообщений в состоянии CharacterEdit.waiting_for_value
- **NUMERIC_FIELDS**: Список полей персонажа, требующих числового ввода

## Требования

### Требование 1: Корректная обработка ошибок ввода

**User Story:** Как пользователь, я хочу получать понятные сообщения об ошибках при неправильном вводе и иметь возможность повторить ввод, чтобы успешно отредактировать поле персонажа.

#### Acceptance Criteria

1. WHEN пользователь вводит некорректное значение для числового поля THEN система SHALL показать сообщение об ошибке и сохранить состояние ожидания ввода
2. WHEN возникает ValueError при обработке ввода THEN система SHALL оставить состояние CharacterEdit.waiting_for_value неизменным
3. WHEN пользователь получает сообщение об ошибке THEN система SHALL продолжать ожидать корректный ввод в том же состоянии
4. WHEN пользователь вводит корректное значение после ошибки THEN система SHALL успешно обновить поле персонажа
5. WHEN отображается сообщение об ошибке THEN оно SHALL быть информативным и указывать на требуемый формат ввода

### Требование 2: Сохранение контекста редактирования

**User Story:** Как пользователь, я хочу, чтобы при ошибке ввода система помнила, какое поле я редактирую, чтобы мне не пришлось начинать процесс заново.

#### Acceptance Criteria

1. WHEN возникает ошибка ввода THEN система SHALL сохранить данные о редактируемом поле в состоянии FSM
2. WHEN пользователь повторяет ввод после ошибки THEN система SHALL использовать сохраненные данные о поле для обновления
3. WHEN происходит ошибка THEN система SHALL сохранить message_id исходного сообщения для последующего редактирования
4. WHEN пользователь вводит корректное значение THEN система SHALL обновить то же поле, которое редактировалось до ошибки

### Требование 3: Улучшенные сообщения об ошибках

**User Story:** Как пользователь, я хочу получать конкретные сообщения об ошибках с указанием ожидаемого формата, чтобы понимать, как правильно ввести значение.

#### Acceptance Criteria

1. WHEN пользователь вводит нечисловое значение для числового поля THEN система SHALL показать сообщение с указанием требуемого формата
2. WHEN отображается сообщение об ошибке THEN оно SHALL содержать информацию о том, какое поле редактируется
3. WHEN пользователь вводит некорректное значение THEN сообщение об ошибке SHALL быть на русском языке
4. WHEN показывается ошибка для поля характеристик THEN сообщение SHALL указывать допустимый диапазон значений (8-20)

### Требование 4: Устойчивость к повторным ошибкам

**User Story:** Как пользователь, я хочу иметь возможность делать несколько попыток ввода, если я продолжаю ошибаться, чтобы в итоге успешно отредактировать поле.

#### Acceptance Criteria

1. WHEN пользователь делает несколько ошибок подряд THEN система SHALL продолжать ожидать корректный ввод
2. WHEN происходят повторные ошибки ввода THEN система SHALL сохранять состояние CharacterEdit.waiting_for_value
3. WHEN пользователь наконец вводит корректное значение THEN система SHALL успешно обработать его независимо от количества предыдущих ошибок
4. WHEN происходят множественные ошибки THEN система SHALL показывать сообщение об ошибке для каждой попытки

### Требование 5: Обработка исключительных ситуаций

**User Story:** Как пользователь, я хочу, чтобы система корректно обрабатывала не только ошибки ввода, но и другие возможные исключения, чтобы бот оставался стабильным.

#### Acceptance Criteria

1. WHEN возникает исключение, отличное от ValueError THEN система SHALL показать общее сообщение об ошибке
2. WHEN происходит неожиданная ошибка THEN система SHALL сбросить состояние на CharacterCreation.editing_sheet для возврата в меню
3. WHEN обработка завершается с ошибкой THEN пользователь SHALL иметь возможность вернуться к редактированию через меню
4. WHEN возникает ошибка базы данных THEN система SHALL показать информативное сообщение пользователю
